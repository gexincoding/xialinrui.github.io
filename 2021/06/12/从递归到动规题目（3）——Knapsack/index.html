从递归到动态规划Problem_3

Knapsack

题目描述

给定两个长度都为N的数组weights和values，weights[i]和values[i]分别代表 i号物品的重量和价值。给定一个正数bag，表示一个载重bag的袋子，装的物品不能超过这个重量。请返回能装下的最大价值。

算法分析

暴力递归

这个道题属于从左往右的尝试模型，遇到每一个位置，只要考虑选还是不选就可以了。定义出递归含义应该不难：process1(int[] w, int[] v, int index, int rest)表示：[0...index-1]决定已经做好，现在剩余容量是rest，请做出从[index...]决策，让价值最大。

其实这道题尝试的basecase的写法很值得探讨。为了方便起见，先放代码：

现在来深入分析一下这个basecase。

  1. 第一个basecase判断rest是否小于零。根据题目要求，如果rest < 0，就不能装了，不是说装进0件货物，而是告诉我们，之前的决策已经失败了，因为已经装不了了。所以这一点就提醒我们，basecase的写法，不能只是顾着递归的含义，而要更加关注上下游的关系以及具体题目所给出的具体内容。比如，这里rest < 0，如果单单从递归含义理解，那么就是遇到了一个不能装东西的背包，返回零好像没有什么问题，但是其实，这还说明了这是一个无效解，我们要用-1去标记一下，因为这样上游就可以判断，我这么做决策，是不是有效的，如果有效的话，我才这么做决策。这就是设置无效解的思想。
  2. 第二个basecase判断，index来到了第一个越界的位置。我们仍然遵循basecase要谨慎分析的原则，理解到，来到越界位置说明之前的决策已经做完了，这个basecase的作用没有别的，只是单纯的返回上一层，返回0不影响结果而已。或者也可以换一种理解方式，0-index-1位置，也就是数组的左右位置，我都做好决策了，而显然，答案是在决策过程中确定的，而不是在最下游basecase收集的，所以可以理解为，目前不需要做决策，所以最大价值就是0，不是说遇到了无效解，而是不需要决策了。

我们来看一下递归的主体：

  1. 我们遇到任何一个普遍的index时候，有两种选择，一是选择不要当前的物品，那么答案显而易见就是process1(w, v, index + 1, rest)；
  2. 二是要这件物品。我们首先用p2接受一下下游的过程，意义是如果要这件物品，后续的最大价值是多少；
  3. 但是p2很有可能是无效解，原因是当前的决策，决策无效了，这样p2就不能成立；
  4. 所以在确定p2真正的值的时候，我们要判断下游是不是有效的，如果无效得0，如果有效，得p2+v[index];
  5. 最后返回p1和p2最大值即可。

动态规划

动态规划主要先是分析好两个范围，寻找出变量有两个，一个是index，一个是rest。index的范围是0-v.length，一共是v.length+1个；rest的范围是0-bag，一共是bag+1个。当然，rest也可能小于零，但是那是无效解，没必要也不可能记录在表格里的。这里的范围是有效解答案的范围。
